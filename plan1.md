# Hindenburg Automator MVP: Comprehensive Implementation Plan

## 1. Project Overview & Architecture

This project is a standalone, local desktop application designed to automate the creation of Hindenburg Narrator session files (`.nhsx`, `Document.xhtml`, `style.css`) for Digital Talking Books (DTBs).

**Technical Stack:**

* **Language:** Python 3.14
* **GUI Framework:** PyQt6 / PySide6
* **Audio Extraction:** `ffprobe` (via `subprocess`)
* **XML Processing:** `xml.etree.ElementTree`
* **API & Security:** `requests` for network calls, `keyring` for secure, OS-native API token storage.
* **Compilation:** PyInstaller (Targeting a lean Windows executable).

**Core Workflow:**

1. **API Fetch & Unpack:** The application fetches a target `.zip` deliverable containing pre-validated, compliant WAV files and unzips them to a local working directory.
2. **Audio Parsing:** Iterates over the WAV files to calculate precise millisecond durations.
3. **Data Generation:** Maps the file naming sequence to XHTML/DAISY structural classes and builds the XML trees.
4. **Session Output:** Writes the synchronized `.nhsx`, `Document.xhtml`, and static `style.css` to the target directory.

---

## 2. User Interface (UI) Design Specifications

The GUI must remain clean, responsive, and oriented around the primary data stream.

* **Top Section:** Input fields for API endpoint queries, optional metadata overrides, and a "Fetch & Generate" trigger button.
* **Center Section:** A `QTestBrowser` (or `QTextBrowser`) widget spanning the primary central real estate of the window. This acts as the main log viewer, displaying API network responses, extraction steps, and XML generation progress.
* **Lower Section:** A distinct `STATUS` text label anchored at the bottom to provide real-time updates of the application state (e.g., "Status: Authenticating...", "Status: Unzipping payload...", "Status: Generating NHSX...").
* **Threading:** All network requests and `ffprobe` subprocesses must be routed through a `QThread` worker to ensure the main GUI loop does not freeze.

---

## 3. Data Extraction & Mapping Logic

The tool relies on the parsed list of sequential audio files. For this MVP, we assume the audio files are perfectly compliant and structured in alphabetical order.

**Timeline Synchronization:**
The tool must maintain a `cumulative_time` variable. As each file is processed, its duration is stored for the `<File>` and `<Region>` tags. The `cumulative_time` before the file's duration is added acts as the `Start` time for the `<Region>` and the `Begin` time for the `<NavPoint>`.

**XHTML ID Generation:**
The script will initialize an integer counter. For every structural text element derived from the audio, it will generate a 5-digit padded string with the prefix `hix` (e.g., `id="hix00001"`).
